# Redaction and Inspection Design

## Overview

Two post-processing subcommands for AI agent workflows:

- `fulgur inspect` — extract structured information from any PDF as JSON
- `fulgur redact` — permanently remove content from a PDF

Typical flow:

```text
fulgur inspect input.pdf > manifest.json
# AI reads manifest, identifies sensitive regions
fulgur redact input.pdf -o redacted.pdf \
  --manifest manifest.json \
  --select "td.personal-info" \
  --text "山田太郎" \
  --regions regions.json
```

---

## fulgur inspect

### Purpose

Extract text positions, embedded images, and document metadata from any PDF as
a single JSON structure. Works on PDFs generated by fulgur or by any other tool.
Primarily useful for AI agents that need to understand PDF content before
deciding what to do with it.

### Output format

```json
{
  "pages": 3,
  "metadata": {
    "title": "Invoice #1234",
    "author": "Acme Corp",
    "creator": "fulgur 0.5.0",
    "created_at": "2026-04-22T10:00:00Z",
    "modified_at": "2026-04-22T10:00:00Z"
  },
  "text_items": [
    {
      "page": 1, "x": 100, "y": 200, "width": 80, "height": 12,
      "text": "山田太郎",
      "font": "NotoSansJP",
      "font_size": 10.0
    }
  ],
  "images": [
    {
      "page": 1, "x": 50, "y": 300, "width": 200, "height": 150,
      "format": "jpeg",
      "width_px": 400,
      "height_px": 300
    }
  ]
}
```

Note: `elements` (CSS selector / tag / class) are only available when the
manifest was generated by `fulgur render --manifest`, because HTML structure is
not preserved in the PDF byte stream.

### CLI

```bash
fulgur inspect <input.pdf> [-o manifest.json]
# default: stdout
```

### Implementation notes

- Text extraction: same content stream parser as the redaction vector engine
  (reuse the same code path)
- Image enumeration: walk page resource dictionaries for `XObject` entries of
  subtype `Image`; extract position from the content stream `Do` operator
- Metadata: read the PDF `Info` dictionary and XMP metadata stream
- Library: `lopdf`

## fulgur redact

`fulgur redact` is a post-processing subcommand that permanently removes content
from a generated PDF. Targeted at AI agent workflows where the agent identifies
what to redact and fulgur handles the actual removal.

Typical flow:

```text
fulgur render input.html -o out.pdf --manifest out.manifest.json
# AI reads manifest, identifies sensitive regions
fulgur redact out.pdf -o redacted.pdf \
  --manifest out.manifest.json \
  --select "td.personal-info" \
  --text "山田太郎" \
  --regions regions.json
```

## Goals

- True redaction: original content deleted from PDF byte stream, not just
  overlaid with a black rectangle
- AI-callable interface: AI agents pass selectors, text patterns, or coordinate
  regions; fulgur does the removal
- Image support: coordinate-specified regions in rasterized areas are blanked via
  re-embedding

## Non-goals

- OCR-based text detection inside images (the caller, typically an AI agent, is
  responsible for detecting image content and supplying coordinates)
- Font subsetting cleanup (unreferenced glyphs after deletion are left in the
  font; this is benign in practice)
- Cosmetic-only redaction (black overlay without content removal)

## Interface

### CLI

```bash
fulgur redact <input.pdf> -o <output.pdf> [options]

Options:
  --manifest <path>     Content manifest from `fulgur render --manifest`
                        (optional; used only as a hint for selector resolution)
  --select <selector>   CSS selector; matched elements are redacted
                        (requires --manifest)
  --text <pattern>      Literal string or /regex/ to match and redact
  --regions <path>      JSON file with explicit bounding boxes (see below)
```

Multiple `--select`, `--text`, and `--regions` flags are additive.

### regions.json

```json
[
  {"page": 1, "x": 100, "y": 200, "width": 150, "height": 30, "type": "vector"},
  {"page": 2, "x": 50,  "y": 100, "width": 200, "height": 40, "type": "image"}
]
```

`type` controls the redaction engine used (see below).

### Manifest sidecar

Generated at render time when `--manifest <path>` is passed to `fulgur render`:

```bash
fulgur render input.html -o out.pdf --manifest out.manifest.json
```

```json
{
  "pages": 3,
  "text_items": [
    {
      "page": 1, "x": 100, "y": 200, "width": 80, "height": 12,
      "text": "山田太郎",
      "selector": "td.name"
    }
  ],
  "elements": [
    {
      "page": 1, "x": 90, "y": 195, "width": 200, "height": 20,
      "tag": "td", "id": null, "classes": ["name"],
      "selector": "table.roster > tbody > tr:nth-child(3) > td.name"
    }
  ]
}
```

**The manifest is a hint, not a guarantee.** Coordinates in the manifest reflect
fulgur's layout intent at render time. Actual redaction coordinates are always
re-derived from the PDF content stream; the manifest is used only to resolve
`--select` selectors to candidate text strings, which are then located precisely
via PDF parsing.

## Redaction Engines

### Vector/text engine (lopdf)

Used for `type: "vector"` regions and all `--text` / `--select` matches.

1. Parse the page content stream into a token sequence.
2. Walk the token sequence tracking the current text matrix (Tm / Td / TD
   operators) to compute each glyph's absolute position.
3. Decode glyph codes to Unicode using the font's ToUnicode map.
4. Identify drawing command blocks (`Tj` / `TJ` / `'` / `"`) whose decoded
   text intersects the target string or bounding box.
5. Remove those drawing commands from the token sequence.
6. Insert a filled black rectangle (`0 g <x> <y> <w> <h> re f`) at the
   vacated position.
7. Reserialize and write the updated content stream back to the page.

`--select` resolution: the manifest maps the selector to candidate text strings.
Those strings are then located in the PDF content stream via step 3–4 above.
The manifest coordinate is never used directly.

**Text matching accuracy note.** Multi-byte CID fonts and split-stream text can
make full Unicode reconstruction difficult. `--text` matching is best-effort for
complex encodings; for guaranteed removal of a known region use `--regions` with
`type: "vector"`.

### Image engine (rasterize + re-embed)

Used for `type: "image"` regions.

1. Rasterize the affected page to a PNG using `pdftocairo` (poppler-utils,
   already required for VRT).
2. Draw a filled black rectangle over the specified region in the PNG.
3. Re-embed the modified PNG as a full-page image object, replacing the
   original page content stream.

This approach trades vector fidelity for simplicity. Pages that contain both
text and image regions may need separate passes: text regions are handled by the
vector engine first, then image regions trigger rasterization of the result.

## Verification

`--verify` re-parses the output PDF after redaction and confirms that target
content no longer appears in the content stream. Exits non-zero if any target is
found, making it safe to use in CI pipelines.

```bash
fulgur redact out.pdf -o redacted.pdf --text "山田太郎" --verify
# → exit 0  if 山田太郎 is absent from the output PDF content stream
# → exit 1  if any target text is still detectable (with stderr diagnostic)
```

### What is checked

| Redaction source | Verification method |
|------------------|---------------------|
| `--text <pattern>` | Re-scan output PDF content stream for the same pattern |
| `--select <selector>` | Re-scan for the text strings resolved from the manifest |
| `--regions type=vector` | Re-scan the bounding box for any surviving text glyphs |
| `--regions type=image` | Verify the original image XObject is no longer referenced (image content is not OCR'd) |

### Scope and limitations

- Verification operates on the PDF content stream only, using the same Unicode
  reconstruction path as the redaction step. If the redaction step could not
  fully decode a glyph sequence, verification has the same blind spot.
- Image regions are verified structurally (image object replaced), not
  semantically (no OCR). Callers requiring OCR-level assurance must use an
  external tool.
- `--verify` does not catch content hidden in PDF metadata, XMP, or embedded
  attachments. Those are out of scope for this subcommand.

## Security Considerations

- Redaction is irreversible: the output PDF has no mechanism to recover the
  removed content.
- The manifest is advisory only; security guarantees rest on PDF stream parsing,
  not manifest coordinates.
- Callers should treat the manifest as untrusted if the PDF and manifest were
  produced separately or by different fulgur versions.
- Font subsetting: deleted glyphs may remain in embedded font programs. This
  does not allow text recovery because the glyph-to-character mapping is also
  removed from the content stream, but sophisticated font extraction tools could
  in theory recover glyph shapes. For highest assurance, pass the output through
  a PDF optimizer that rebuilds font subsets.

## Implementation Notes

- PDF library: `lopdf` (already preferred in the project; pure Rust, no C
  dependencies, WASM-compatible)
- Rasterization: `pdftocairo` (poppler-utils; already used in `fulgur-vrt`)
- The redact subcommand lives in `crates/fulgur-cli`; the core redaction logic
  lives in `crates/fulgur` as a `redact` module so library users can call it
  directly
- Manifest generation hooks into the existing render pipeline at the point where
  Pageable boxes are finalized (after pagination, before PDF emission)
