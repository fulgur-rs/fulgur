# Phase 4 Pageable Inventory

Date: 2026-04-30
Issue: fulgur-9t3z (Phase 4 prerequisite — feeds architecture brainstorm)

Purpose: classify each concrete Pageable implementor by its `wrap()` and `draw()` responsibilities, so the upcoming Phase 4 redesign (delete Pageable trait, geometry-driven render) has a complete migration target list.

## Findings

- **Trivially flattenable types**: `SpacerPageable` (no draw at all), `ImagePageable`, `SvgPageable`, and all five marker types (`BookmarkMarker`, `StringSet`, `RunningElementMarker`, `CounterOpMarker`, plus the conceptually-empty wrapper bodies for the latter three). These map cleanly to either no DrawOp at all (markers feed a separate metadata table) or a single self-contained `DrawOp { rect, payload }`. The marker wrappers (`StringSet`/`RunningElement`/`CounterOp`Wrapper) only delegate at draw time; their entire semantic load is in `slice_for_page`'s first-page-only logic, which can move into a per-fragment "first appearance" flag on the geometry table.

- **Hardest types, ranked top 3**:
  1. **`ParagraphPageable`** — line-internal state (per-glyph `x_offset` / `x_advance`, decoration spans, link spans, vertical-aligned inline images, inline-box children that are themselves full Pageable subtrees) is not captured by any current geometry record, only by the line band's top/height. Either flatten lines into a self-contained "shaped-line draw stream" leaf op, or keep `ShapedLine` as the payload of a `DrawOp::Paragraph` and accept a recursion edge for inline-box content.
  2. **`BlockPageable`** — backgrounds, gradients (linear / radial / conic), box-shadows, border styles (solid/dashed/dotted/double/groove/ridge/inset/outset), border-radius, and overflow-clip path generation collectively emit a huge amount of geometry per node. Each is currently composed dynamically inside `draw()`; flattening means either pushing all of `background::*` and `draw_block_border` into a DrawOp serializer, or keeping `BlockStyle` as a side-channel attached to the geometry record (preferred — much smaller diff).
  3. **`TablePageable`** — per-page header repetition + body cell rebase logic is the most "knows about pagination" of any concrete leaf. The current `slice_for_page` clones `header_cells` and shifts body `pc.y` so the first kept body sits at `header_height`. Phase-4 needs the geometry record to expose "table fragment on page p has cells {header_clones, body_subset}" with the rebase already applied, otherwise table headers won't repeat.

- **Wrapper composition**: The wrappers are *almost* independent. `BookmarkMarkerWrapper`, `StringSetWrapper`, `RunningElementWrapper`, and `CounterOpWrapper` are functionally interchangeable in their `slice_for_page` (all use `is_first_page_for` on the inner's `node_id`) and their `draw` is a pure delegate (with the bookmark wrapper additionally calling `marker.draw` for the BookmarkCollector side-effect). They do **not** share state — each holds its own `markers`/`ops` vec independently — and they nest cleanly. **`TransformWrapperPageable` is the exception**: it pushes/pops both a krilla surface transform and a `LinkCollector` transform stack, which means any link-rect emitted *inside* a transform must be transformed at emit time. This is the only wrapper where the order-of-composition with `BookmarkMarker`/`StringSet`/etc. wrappers matters for correctness. Convert order today is roughly Block → Bookmark → CounterOp → StringSet → RunningElement → Transform (innermost first), so the transform sits closest to the inner content and link-rect transformation works.

- **`wrap()` side effects that would be lost**: only one — `BlockPageable::wrap` lazily calls `child.wrap(avail_width, 10000.0)` for any child whose `height() == 0` (these are split-fragment children that haven't been re-wrapped after `split()`). With Phase-4 deleting the trait, this lazy re-wrap goes away naturally because (a) splits become per-page slices driven entirely by the geometry table, and (b) every node's size is materialized in the geometry table at convert/layout time. No persistent state escapes `wrap()` other than `cached_size` / `cached_height` / `layout_size`, all of which are read-back caches. `ParagraphPageable::wrap` only sums line heights — derivable from geometry. None of the wrappers do anything in `wrap()` except delegate.

- **Things `draw()` does that the fragmenter geometry doesn't capture today**:
  - Per-glyph horizontal advance and per-line decoration spans (paragraph) — geometry only knows the line band.
  - Link rects emitted into `Canvas::link_collector`. These currently come from glyph-run width sums, image rects, inline-box rects — all line-internal in the paragraph case. The geometry table has no concept of "link span".
  - Bookmark Y-position recorded into `BookmarkCollector` at draw time (the `y` argument is the first-fragment top). Geometry knows fragment top, but the bookmark→outline pipeline currently goes through `Canvas`, not through geometry.
  - `TransformWrapper`'s push/pop on `LinkCollector::transform_stack` (mutates link rect coords lazily). Geometry has node-level transforms but doesn't track the transform stack as link rects accumulate.
  - `MulticolRulePageable::draw` paints rules whose vertical extent is `min(h_left, h_right)` over per-`ColumnGroup` data. The column-group geometry exists (`MulticolGeometryTable`) but is only consulted at draw time via this wrapper.
  - `BlockPageable`'s overflow clip path is computed at draw time from `style + (x, y, w, h)`. Geometry knows the box but not "clip vs no-clip"; that's a `BlockStyle` attribute.

These are the responsibilities Phase-4 must absorb into either (a) richer per-node entries in `PaginationGeometryTable`, (b) a new "draw-op stream" emitted alongside geometry at convert time, or (c) a small set of side-channel state collectors (links, bookmarks, string-set values, counter ops) keyed by node_id × page_index instead of mutated through `Canvas` during draw.
